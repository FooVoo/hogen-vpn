# VLESS+Reality — Technical Reference

This document explains how VLESS+Reality works and documents the manual parameter requirements. In this project, everything described here is handled automatically by `generate-secrets.sh` and `render-xray-config.sh`. Refer to `setup.md` for the actual deployment procedure.

## What VLESS+Reality does

VLESS+Reality routes **all traffic** through your VPS, disguising it as HTTPS to a real website. The server presents a genuine TLS handshake to a chosen "cover domain" — someone probing the server without your credentials sees a legitimate TLS session, not a proxy.

Key properties:
- No server certificate required (borrows the cover domain's TLS)
- TLS 1.3 + HTTP/2, indistinguishable from the cover domain to passive observers
- Client requires: UUID, X25519 public key, short ID, SNI, `fp=chrome`

## Parameters

### UUID (user identity)

Generated by `generate-secrets.sh` via `/proc/sys/kernel/random/uuid` or `python3 -c "import uuid; print(uuid.uuid4())"`.

```bash
cat /proc/sys/kernel/random/uuid
```

### X25519 keypair

Generated using Xray itself:

```bash
docker run --rm ghcr.io/xtls/xray-core:26.3.27 xray x25519
```

Output:
```
Private key: 2KZ4uouMKgI8nR-LDJNP1_MHisCJOmKGj9jUjZLncVU
Public key:  Z84J2IelR9ch3k8VtlVhhs5ycBUlXA7wHBWcBrjqnAw
```

**Private key** stays on the server only. **Public key** goes into every client config.

### ShortId

8 random hex bytes:

```bash
openssl rand -hex 8
# example: 6ba85179e30d4fc2
```

### Cover domain (SNI / dest)

The Reality cover domain must support **TLS 1.3** and **HTTP/2**. It must not redirect.

In this project, `generate-secrets.sh` selects from a 35-domain pool (see `XRAY_COVER_DOMAINS` in `.env`). The active domain rotates every 120 minutes via a systemd timer.

Verify a candidate domain from the VPS:

```bash
curl -vI --http2 --tlsv1.3 https://www.microsoft.com 2>&1 | grep -E "TLSv1.3|HTTP/2"
```

Both `TLSv1.3` and `HTTP/2` must appear in the output.

## Xray server config structure

`xray/config.json` (as rendered by `render-xray-config.sh`) contains two inbounds:

```json
{
  "log": { "loglevel": "warning" },
  "inbounds": [
    {
      "listen": "0.0.0.0",
      "port": 8443,
      "protocol": "vless",
      "settings": {
        "clients": [{ "id": "<UUID>", "flow": "xtls-rprx-vision" }],
        "decryption": "none"
      },
      "streamSettings": {
        "network": "tcp",
        "security": "reality",
        "realitySettings": {
          "show": false,
          "dest": "<SNI>:443",
          "serverNames": ["<SNI>"],
          "privateKey": "<XRAY_PRIVATE_KEY>",
          "shortIds": ["<XRAY_SHORT_ID>"],
          "maxTimeDiff": 60000
        }
      },
      "sniffing": { "enabled": true, "destOverride": ["http", "tls", "quic"] }
    },
    {
      "listen": "0.0.0.0",
      "port": 8388,
      "protocol": "shadowsocks",
      "settings": {
        "method": "2022-blake3-aes-256-gcm",
        "password": "<SS_PASSWORD>",
        "network": "tcp,udp"
      }
    }
  ],
  "outbounds": [
    { "protocol": "freedom", "tag": "direct" },
    { "protocol": "blackhole", "tag": "block" }
  ]
}
```

## VLESS URI format

```
vless://<UUID>@<SERVER_IP>:8443?encryption=none&flow=xtls-rprx-vision&security=reality&sni=<SNI>&fp=chrome&pbk=<PUBLIC_KEY>&sid=<SHORT_ID>&type=tcp#MyVPN
```

This URI is generated and stored in `VLESS_URI` by `generate-secrets.sh`, and rotated automatically when the cover domain changes.

## NTP requirement

Reality validates timestamps. If the server clock drifts more than ~60 seconds, clients silently fail to connect.

```bash
timedatectl status   # must show: NTP synchronized: yes
timedatectl set-ntp true   # enable if not already active
```

## Rotation behavior

After a cover domain rotation:

- `xray/config.json` is rewritten with the new `dest` and `serverNames`
- `VLESS_URI` in `.env` is updated with the new SNI
- `XRAY_ROTATION_MESSAGE` is updated on the credentials page
- Xray container is restarted
- **Previously imported VLESS profiles stop working** — clients must re-import from the credentials page

Shadowsocks on port 8388 is unaffected by rotation (different inbound, no cover domain).

## Troubleshooting

**Clients connect but traffic doesn't flow:**
```bash
timedatectl status   # check NTP sync
docker compose logs xray   # check for errors
```

**Container won't start:**
```bash
docker compose logs xray
./render-xray-config.sh   # regenerate config from .env
docker compose restart xray
```

**Port 8443 unreachable:**
```bash
ufw status
ss -tlnp | grep 8443
# Also check the hosting provider's network-level firewall
```

**`serverNames` mismatch (client TLS error):**
```bash
echo | openssl s_client -connect www.microsoft.com:443 2>/dev/null \
  | openssl x509 -noout -text | grep "DNS:"
# Verify the cert covers the exact name in serverNames
```
